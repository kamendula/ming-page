# 补明史传记网站 - 设计方案（简单初始版）

## 📋 项目概述

这是一个用于阅读明代人物传记的静态网站，支持从 Word 文档自动转换为在线阅读页面。

**部署平台**：GitHub Pages（免费、零配置、与 GitHub 深度集成）

### 为什么选择 GitHub Pages？

✅ **完全免费** - 无需任何费用  
✅ **零配置** - 只需在 Settings 中点几下  
✅ **自动部署** - 推送代码自动更新网站  
✅ **免费 HTTPS** - 自动配置 SSL 证书  
✅ **自定义域名** - 支持绑定自己的域名  
✅ **无需注册其他服务** - 只用 GitHub 一个账号  
✅ **访问地址** - `https://你的用户名.github.io/Ming-Page/`

| 对比项 | GitHub Pages | CloudFlare Pages | Vercel/Netlify |
|--------|--------------|------------------|----------------|
| 注册账号 | ✅ 已有 GitHub | ❌ 需要单独注册 | ❌ 需要单独注册 |
| 配置难度 | ⭐ 极简（2步） | ⭐⭐ 简单 | ⭐⭐ 简单 |
| 部署速度 | 30秒 - 1分钟 | 20-40秒 | 20-40秒 |
| 自定义域名 | ✅ 免费 | ✅ 免费 | ✅ 免费 |
| HTTPS | ✅ 自动 | ✅ 自动 | ✅ 自动 |
| 最适合 | 个人项目/文档 | 任何项目 | 动态应用 |

**结论**：对于静态文档网站，GitHub Pages 是最简单的选择！

## 二、工作流程

### 你的日常操作流程

```bash
1. 在 Word 中编写/修改传记 → 保存到 doc/ 文件夹
2. git add doc/新传记.docx
3. git commit -m "添加新传记"
4. git push
5. 等待 1-2 分钟 → 访问 https://你的用户名.github.io/Ming-Page/ 
6. 网站自动更新 ✅
```

**就这么简单！** 你只需要专注于编写内容，技术细节全自动。

### 自动化流程（背后发生的事）

```
GitHub 监听到 doc/ 文件夹有变化
    ↓
触发 GitHub Actions
    ↓
运行转换脚本 scripts/convert.js
    ↓
读取所有 .docx 文件
    ↓
使用 mammoth.js 提取文本内容
    ↓
解析：标题、作者、日期、正文、注释
    ↓
生成 JSON 文件到 public/data/
    ↓
自动提交 JSON 文件到仓库
    ↓
GitHub Pages 检测到 public/ 更新
    ↓
自动重新部署网站 🎉
```

## 三、Word 文档格式约定

### 推荐格式示例

```
陈山传                           ← 第一行作为标题

　　陈山，福建沙县人，元至正二十二年生。

　　其先世居长葛，后徙南剑州沙县。天祖世卿...

　　永乐六年，山以奉化县学教谕被荐于朝...

【1】《赠资善大夫户部尚书兼谨身殿大学士陈公》：陈始自长葛徙南剑之沙县...
【2】《太宗文皇帝实录》：壬戌年，升任翰林院修撰吴溥...
【3】...

——乙巳蛇年农历十月初九关山补写
```

### 自动识别规则

| 元素 | 识别规则 | 示例 |
|------|---------|------|
| **标题** | 第一行非空文本 | `陈山传` |
| **传主** | 标题去掉"传"字 | `陈山` |
| **正文** | 从第一段到第一个【数字】前 | 所有段落 |
| **注释** | 【1】【2】等标记的内容 | `【1】《史料》：内容...` |
| **作者** | 末尾"——"行提取 | `关山` |
| **日期** | 末尾"——"行提取 | `乙巳蛇年农历十月初九` |

### 注意事项

✅ **支持**：
- 正常段落文字
- 注释标记【1】【2】...【99】
- 书名号《》内的引文来源
- 传统署名格式

⚠️ **暂不支持**（初始版）：
- Word 中的表格
- 图片
- 复杂格式（颜色、字体等）
- 批注

## 四、JSON 数据格式

### index.json（文档索引）

```json
{
  "total": 3,
  "updatedAt": "2025-12-31T12:00:00.000Z",
  "documents": [
    {
      "title": "陈山传",
      "subject": "陈山",
      "author": "关山",
      "date": "乙巳蛇年农历十月初九",
      "file": "陈山.json"
    },
    {
      "title": "李思孝传",
      "subject": "李思孝",
      "author": "关山",
      "date": "",
      "file": "李思孝.json"
    }
  ]
}
```

### 单个传记 JSON（如 陈山.json）

```json
{
  "title": "陈山传",
  "subject": "陈山",
  "author": "关山",
  "date": "乙巳蛇年农历十月初九",
  "content": "　　陈山，福建沙县人，元至正二十二年生。\n\n　　其先世居长葛...",
  "notes": [
    {
      "id": 1,
      "content": "《赠资善大夫户部尚书兼谨身殿大学士陈公》：陈始自长葛...",
      "source": "赠资善大夫户部尚书兼谨身殿大学士陈公"
    },
    {
      "id": 2,
      "content": "《太宗文皇帝实录》：壬戌年，升任翰林院修撰...",
      "source": "太宗文皇帝实录"
    }
  ],
  "wordCount": 1234,
  "sourceFile": "陈山传.docx",
  "updatedAt": "2025-12-31T12:00:00.000Z"
}
```

**字段说明**：
- `wordCount`: 正文字数统计
- `notes[].source`: 注释来源书名（用于右侧导航显示，提取《》内容）

## 十、智能功能实现逻辑

### 人物名称提取

**简单规则（第一版）**：
1. 提取二字/三字中文人名
2. 过滤常见非人名词汇
3. 统计出现频率，取前 10 位
4. 保留带有明确人物标识的词：
   - X帝、X王、X公、X侯
   - 职务后的人名（如"尚书XXX"）

**示例正则**：
```javascript
// 提取可能的人名
const namePattern = /([^\u4e00-\u9fa5]|^)([\u4e00-\u9fa5]{2,3})(?=帝|王|公|侯|曰|对曰|言)/g;
```

### 相关传记推荐算法

**推荐逻辑**：
1. **直接关联**：当前传记中提到的人物，如果有传记则优先推荐
2. **共同人物**：两篇传记共同提到的人物越多，相关度越高
3. **相同作者**：同一作者撰写的其他传记

**计算公式**：
```
相关度分数 = 
  共同人物数 × 3 + 
  (相同作者 ? 2 : 0) + 
  (被提及 ? 5 : 0)
```

**显示规则**：
- 显示相关度最高的 3-5 篇
- 至少有 1 个共同人物才显示

## 十一、部署步骤（GitHub Pages）

### 方案选择

GitHub Pages 提供三种发布方式，**推荐方案 1**（最简单）：

#### **方案 1：从 main 分支的 /public 文件夹发布**（推荐 ⭐）

**优点**：最简单，无需额外配置

**步骤**：
1. 进入 GitHub 仓库 Settings
2. 左侧点击 **Pages**
3. Source 选择：`Deploy from a branch`
4. Branch 选择：`main` → 文件夹选择 `/public` → Save
5. 等待 1-2 分钟，访问 `https://your-username.github.io/Ming-Page/`

**工作流程**：
```
你推送 Word 文档到 doc/
    ↓
GitHub Actions 自动转换生成 JSON 到 public/data/
    ↓
GitHub Actions 自动提交 public/ 文件夹
    ↓
GitHub Pages 自动检测并部署
    ↓
网站更新完成 ✅
```

#### **方案 2：使用 gh-pages 分支**（适合进阶用户）

**优点**：main 分支保持干净，部署内容独立

**步骤**：
1. GitHub Actions 自动创建 `gh-pages` 分支
2. Settings → Pages → Source 选择 `gh-pages` 分支
3. 部署内容单独管理

#### **方案 3：使用 GitHub Actions 直接部署**

**优点**：最灵活，可以添加构建步骤

**步骤**：使用 `actions/deploy-pages` action

---

### GitHub Actions 配置（必需）

需要确保 GitHub Actions 有写入权限：

1. 进入仓库 **Settings**
2. 左侧点击 **Actions** → **General**
3. 找到 **Workflow permissions**
4. 选择 **"Read and write permissions"**
5. 勾选 **"Allow GitHub Actions to create and approve pull requests"**（可选）
6. 点击 **Save**

### Google AdSense 接入步骤

1. **申请 AdSense 账号**
   - 访问 [https://www.google.com/adsense](https://www.google.com/adsense)
   - 使用 Google 账号登录
   - 提交网站审核

2. **获取广告代码**
   - 审核通过后，进入 AdSense 后台
   - 创建广告单元（推荐尺寸：300×250、300×600）
   - 获取客户端 ID (ca-pub-XXXXXXXXXXXXXXXX)
   - 获取广告位 ID

3. **修改配置文件**
   - 编辑 `public/config.js`
   - 填入你的 AdSense 客户端 ID
   - 填入各广告位的 ID
   - 设置 `ads.enabled = true`

4. **添加 AdSense 脚本**（自动集成在 HTML 中）
   ```html
   <script async src="https://pagead2.googlesyndication.com/pagead/js/adsbygoogle.js?client=ca-pub-XXXXXXXXXXXXXXXX"
        crossorigin="anonymous"></script>
   ```

5. **广告位代码示例**
   ```html
   <ins class="adsbygoogle"
        style="display:block"
        data-ad-client="ca-pub-XXXXXXXXXXXXXXXX"
        data-ad-slot="1234567890"
        data-ad-format="auto"></ins>
   <script>
        (adsbygoogle = window.adsbygoogle || []).push({});
   </script>
   ```

### 广告展示策略

- **桌面端**：右侧栏显示两个广告位
- **平板端**：只显示一个广告位（顶部）
- **移动端**：切换为横幅广告或不显示
- **广告加载失败**：显示自定义推广内容（备用方案）

## 十六、注意事项

### Word 文档处理

- ⚠️ 确保 Word 文档是 `.docx` 格式（不是 `.doc`）
- ⚠️ 临时文件（以 `~` 开头）会被自动跳过
- ⚠️ 注释编号必须从【1】开始，连续递增

### Git 仓库管理

- ✅ `doc/` 文件夹提交 Word 原文件
- ✅ `public/data/` 文件夹提交生成的 JSON（由 Actions 自动提交）
- ⚠️ 不要手动编辑 `public/data/` 中的 JSON（会被覆盖）

### 性能考虑

- 当前方案适合 100 篇以内的传记
- 如果超过 100 篇，可能需要考虑分页或懒加载
- 每个 JSON 文件建议不超过 500KB
